## ✅ 核心功能

### 1. LLM 响应分析日志

**位置**: `agent.py` - `run()` 方法  
**记录内容**:

```python
[Iteration X] LLM 响应分析:
  - 响应内容预览：{前 500 字符}
  - 工具调用：{tool_name(args)} 或 ⚠️ 无工具调用
  - 响应质量分析：
    * ✅ 正常 / ❌ 严重问题 / ⚠️ 警告
  - 系统提示词验证：
    * ✅ CRITICAL RULES 存在
    * ✅ 工具使用说明存在
  - 用户消息提醒验证：
    * ✅ SYSTEM REMINDER 存在
    * ✅ 具体工具名称存在
```

**诊断价值**: 
- ✅ 判断模型是否生成工具调用
- ✅ 验证系统提示词是否正确传递
- ✅ 确认提醒机制是否生效

### 2. 流式调用详细日志

**位置**: `agent.py` - `_stream_invoke()` 方法  
**记录内容**:

```python
[流式调用] 开始流式调用 LLM
  - 消息历史长度：X
  - 最后消息类型：HumanMessage/AIMessage
  - 最后内容预览：{前 200 字符}

[流式 chunk #X] 检测到工具调用片段:
  - 工具名称：xxx
  - 参数预览：{...}
  - ID: call_xxx

[流式调用] 完成统计:
  - 总 chunk 数：X
  - 收集文本长度：X 字符
  - 检测到工具调用：是/否
```

**诊断价值**:
- ✅ 追踪流式调用过程
- ✅ 检测工具调用是否分散在多个 chunk 中
- ✅ 验证混合模式是否必要

### 3. 消息历史统计日志

**位置**: `agent.py` - `run()` 方法  
**记录内容**:

```python
[消息历史统计]
  - 总消息数：X
  - 历史工具调用：#2: list_directory, #4: read_file
  - 或 ⚠️ 历史消息中无任何工具调用
```

**诊断价值**:
- ✅ 了解整体对话上下文
- ✅ 追踪工具调用历史
- ✅ 判断是否是持续性不调用工具

## 📝 实际案例分析

### 案例 1: 成功调用工具

**日志片段**:
```
2026-03-07 12:19:38 [INFO] Agent: [Iteration 1] LLM 响应分析:
2026-03-07 12:19:38 [DEBUG] Agent:   - 响应内容预览：(空内容)
2026-03-07 12:19:38 [INFO] Agent:   - 工具调用：list_directory({'path': '.'})
2026-03-07 12:19:38 [DEBUG] Agent:   - ✅ 工具调用成功，无需重试
```

**分析**:
- ✅ 第一次迭代就成功调用工具
- ✅ 响应内容为空（正常，纯工具调用）
- ✅ 参数完整：{'path': '.'}

**结论**: 模型响应正常，提示词有效

---

### 案例 2: 提示词正常但模型不调用工具

**日志片段**:
```
2026-03-07 12:19:45 [WARNING] Agent: [Iteration 3] LLM 响应分析:
2026-03-07 12:19:45 [WARNING] Agent:   - ⚠️ 无工具调用
2026-03-07 12:19:45 [WARNING] Agent:   - ⚠️ 警告：用户请求需要工具，但模型未调用
2026-03-07 12:19:45 [DEBUG] Agent:   - ✅ 系统提示词包含 CRITICAL RULES
2026-03-07 12:19:45 [DEBUG] Agent:   - ✅ 系统提示词包含工具使用说明
2026-03-07 12:19:45 [DEBUG] Agent:   - ✅ 用户消息包含 SYSTEM REMINDER
2026-03-07 12:19:45 [DEBUG] Agent:   - ✅ 提醒中包含具体工具名称
```

**分析**:
- ✅ 系统提示词完整（包含 CRITICAL RULES 和工具使用说明）
- ✅ 用户消息提醒也完整
- ❌ 但模型仍然不调用工具

**结论**: **模型响应问题**，不是提示词问题

**可能原因**:
1. 模型温度过高（建议检查 temperature 设置）
2. 模型本身倾向于聊天而非行动
3. 可能需要 Few-shot 示例

**解决方案**:
- 调整 temperature（从 0.7 降到 0.5）
- 添加成功的工具调用示例到提示词
- 考虑更换更适合工具调用的模型

---

### 案例 3: 流式调用未检测到工具

**日志片段**:
```
2026-03-07 12:19:49 [DEBUG] Agent: [流式调用] 开始流式调用 LLM
2026-03-07 12:19:49 [DEBUG] Agent:   - 消息历史长度：6
2026-03-07 12:19:49 [DEBUG] Agent:   - 最后消息类型：AIMessage

2026-03-07 12:19:49 [INFO] Agent: [流式调用] 完成统计:
2026-03-07 12:19:49 [INFO] Agent:   - 总 chunk 数：95
2026-03-07 12:19:49 [INFO] Agent:   - 收集文本长度：526 字符
2026-03-07 12:19:49 [INFO] Agent:   - 检测到工具调用：否
```

**分析**:
- ✅ 流式调用正常执行
- ✅ 收集了大量文本（526 字符）
- ❌ 未检测到工具调用

**结论**: **正常现象**

**解释**:
- LangChain 的流式 API 会将工具调用分散在多个 chunk 中
- 这是已知限制，不是 bug
- 已通过混合模式解决：流式显示 + 重新 invoke 获取完整工具信息

**验证**: 检查后续是否有完整的工具调用日志

## 📊 日志关键字段速查

### LLM 响应分析部分

| 日志 | 含义 | 正常值 |
|------|------|--------|
| `响应内容预览` | AI 返回的文本 | 有意义的内容或空 |
| `工具调用：xxx` | 调用的工具列表 | 应该有工具名 |
| `⚠️ 无工具调用` | 警告标记 | 不应出现 |
| `❌ 严重问题` | 错误标记 | 绝不应出现 |

### 系统提示词验证部分

| 日志 | 含义 | 期望结果 |
|------|------|----------|
| `系统提示词长度` | 提示词字符数 | >500 |
| `✅ CRITICAL RULES` | 包含关键规则 | 应该有 ✅ |
| `✅ 工具使用说明` | 包含工具使用指导 | 应该有 ✅ |
| `❌ 缺少...` | 缺失关键内容 | 不应出现 ❌ |

### 流式调用部分

| 日志 | 含义 | 说明 |
|------|------|------|
| `chunk 数` | 流式片段数量 | 通常 5-20 |
| `收集文本长度` | 文本总长度 | 可变 |
| `检测到工具调用` | 是否检测到 | 是/否 |

## 💡 常见问题与解决方案

### Q1: 如何区分是模型问题还是提示词问题？

**A**: 查看系统提示词验证日志：

```
- ✅ CRITICAL RULES
- ✅ 工具使用说明
```

- 如果都有 ✅ 但模型仍不调用 → **模型问题**
- 如果有 ❌ 或缺失 → **提示词问题**

### Q2: Temperature 应该设置为多少？

**A**: 
- 工具调用场景：建议 **0.5-0.7**
- 创造性任务：可以更高 **0.7-0.9**
- 当前配置：检查 `config/config.json`

### Q3: 流式调用总是检测不到工具怎么办？

**A**: 
- ✅ 这是正常的，LangChain 流式 API 的限制
- ✅ 已通过混合模式解决（流式 + 完整调用）
- ✅ 不影响最终工具调用成功率

### Q4: 如何查看完整的调试信息？

**A**:
```python
from src.learn_agent.logger import set_log_level
import logging

set_log_level(logging.DEBUG)
```

或在诊断工具中自动设置。
